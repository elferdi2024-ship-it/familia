1. Resumen Ejecutivo
MiBarrio.uy v7.0 es un proyecto inmobiliario ambicioso y bien concebido, construido sobre un stack moderno (Next.js, Firebase, Algolia, Tailwind CSS 4) con un ritmo de desarrollo muy alto para un equipo unipersonal. Sin embargo, el autodiagnóstico que acompaña el código presenta optimismo sistemático que puede generar falsas certezas antes del lanzamiento a producción real con usuarios finales.
Esta auditoría independiente identifica cuatro hallazgos críticos y ocho hallazgos de alta prioridad que deben resolverse antes o inmediatamente después del go-live. Los puntos fuertes del proyecto son genuinos y merecen reconocimiento; los gaps, sin embargo, representan riesgos reales de producción.

CRÍTICO-3: Rate Limiting In-Memory — Inefectivo en Serverless
El middleware documenta rate limiting activo, pero si la implementación usa variables en memoria (Map/objeto JavaScript), es inefectivo en producción serverless de Vercel. Cada invocación de función tiene su propia memoria; los contadores se resetean con cada cold start. El documento menciona Upstash KV, lo que es la solución correcta — pero no queda claro si el rate limiting del middleware usa Upstash o memoria local.
Verificación requerida:
 Revisar el código de middleware.ts (líneas 1-102). Confirmar que MAX_REQUESTS_API y MAX_REQUESTS_LEADS se almacenan y verifican contra Upstash Redis, no contra un Map en memoria. Si es en memoria, el rate limiting no funciona.
Evidencia de riesgo:
 El propio roadmap menciona &apos;Activar Rate Limiting (1 min)&apos; como tarea pendiente en versiones anteriores, lo que sugiere que inicialmente era en memoria y Upstash se integró después. Confirmar que la integración es completa.

CRÍTICO-4: Prisma Schema sin Base de Datos Activa — Deuda Arquitectural
Existe un prisma/schema.prisma en el proyecto pero la documentación lo marca como &apos;futuro&apos;. Esto crea una ambigüedad arquitectural significativa: el proyecto usa Firebase/Firestore como base de datos principal, pero tiene una capa Prisma durmiente.
Riesgos:
 (1) Si algún desarrollador asume que Prisma está activo y escribe código contra él, los datos no se persistirán. (2) La estrategia de migración de Firestore a PostgreSQL no está definida. (3) El schema de Prisma puede estar desincronizado con el modelo de datos real de Firestore.
Decisión requerida:
 O eliminar Prisma del proyecto hasta que sea necesario, o documentar explícitamente la estrategia de migración y el timeline. Dejar un schema durmiente en producción es una trampa para futuros contribuidores.
3. Hallazgos de Alta Prioridad
ALTA-1: Sentry en 100% Sample Rate en Producción
El proyecto configura Sentry con tracesSampleRate: 1.0 (100%) para performance tracing. En producción real, esto significa que cada request se traza completamente. Con tráfico moderado (1,000 usuarios/día), esto puede agotar rápidamente el plan gratuito de 5K eventos/mes y generar costos.
Configuración recomendada para producción:
 tracesSampleRate: 0.1 (10%), replaysSessionSampleRate: 0.05 (5%), replaysOnErrorSampleRate: 1.0 (mantener al 100% — apropiado para errores).

ALTA-2: Feed Social sin Moderación de Contenido
La Red Social Inmobiliaria permite a agentes publicar posts con texto e imágenes. No hay documentación de un sistema de moderación, filtrado de spam, o límites de contenido en el feed. En un contexto B2B/B2C inmobiliario, el riesgo de publicaciones inapropiadas, spam, o competencia desleal entre agentes es real y puede dañar la reputación de la plataforma.
Recomendación:
 Implementar al mínimo: (1) Firestore rules que limiten la frecuencia de posts por usuario (máx. 5 posts/día), (2) validación de longitud mínima y máxima del contenido, (3) flag de reporte por parte de otros usuarios antes del go-live con múltiples agentes.

ALTA-3: Solo 2 Rutas API para 38 Rutas Frontend
El proyecto documenta 38 rutas de frontend pero solo 2 API routes (/api/properties y /api/search/suggestions). Esto sugiere que la mayoría de las operaciones de escritura se realizan directamente contra Firebase desde el cliente. Esta arquitectura tiene implicaciones de seguridad:

(2) Operaciones como la publicación de propiedades que deberían incluir validación server-side (ej. límite de propiedades por plan) no tienen ese control. (3) El sync con Algolia (npm run algolia:sync) parece ser un script manual, no un trigger automático.
Recomendación:
 Agregar al menos 3 API routes adicionales: /api/feed/post (creación de posts con validación server), /api/properties/publish (con validación de límites de plan), y /api/algolia/sync (webhook trigger post-publicación).

ALTA-4: PWA sin Service Worker — No Hay Capacidad Offline Real
La documentación declara PWA al 85% pero sin service worker para offline. Un manifest.json + iconos hace que la app sea instalable, pero no es una PWA funcional en el sentido de Progressive Web App. Los usuarios en conexiones lentas (muy común en Uruguay fuera de Montevideo) no tienen experiencia offline ni stale-while-revalidate.
Impacto:
 Para agentes inmobiliarios mostrando propiedades in situ (con conectividad variable), la falta de offline es un bloqueador de UX real. La competencia InfoCasas tampoco tiene esto, lo que es una oportunidad de diferenciación desaprovechada.

ALTA-5: Algolia Sync Manual — Inconsistencia de Datos Potencial
El script npm run algolia:sync sugiere que la sincronización entre Firestore y el índice de Algolia es manual. Si un agente publica una propiedad y el sync no se ejecuta, la propiedad no aparecerá en los resultados de búsqueda instantánea hasta el próximo sync manual.
Solución:
 Implementar un Cloud Function de Firebase que escuche los eventos onCreate/onUpdate/onDelete de la colección properties y sincronice automáticamente con Algolia. Esto elimina la dependencia del sync manual y garantiza consistencia en tiempo real.

ALTA-6: Blog sin CMS — Limitación de Escalabilidad Editorial
Los artículos del blog se almacenan en /data/posts.ts como código TypeScript. Agregar o editar un artículo requiere un deployment completo. Para el objetivo de 10+ artículos/mes del roadmap, esto significa que el equipo de contenido necesita acceso al repositorio y capacidad de deployment.
Opciones:
 (1) Corto plazo: mantener posts.ts pero agregar validación de frontmatter. (2) Mediano plazo: migrar a Contentlayer/MDX files en el repositorio (sin CMS externo, pero con hot reload). (3) Largo plazo: Sanity.io o Contentful para que marketing pueda publicar sin código.

ALTA-7: /api/search/suggestions sin Validación Zod
La tabla de API routes documenta que /api/properties tiene validación Zod pero /api/search/suggestions no. Un endpoint de búsqueda sin validación es vulnerable a queries extremadamente largas, caracteres especiales, e inyecciones de caracteres especiales de Algolia.
Fix:
 Agregar schema Zod mínimo: z.object({ q: z.string().min(1).max(100).trim() }). Es una corrección de 5 líneas con impacto de seguridad real.

ALTA-8: Contextos Potencialmente Sobredimensionados
El proyecto documenta 5 React contexts para 120+ archivos. Sin ver el código directamente, la proporción sugiere que algunos contextos pueden tener demasiadas responsabilidades. Contextos que mezclan estado de UI con estado de datos son una fuente común de re-renders innecesarios en aplicaciones Next.js con React 19.
Verificación recomendada:
 Revisar si algún context incluye tanto estado del usuario como propiedades de UI (ej. modal open/closed + user data). Si es así, separar los concerns. Con React Compiler habilitado esto es menos crítico, pero es buena práctica.
